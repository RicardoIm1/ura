2. DISEÑO DE ALGORITMOS

2.1 Conceptos básicos

Para empezar a diseñar algoritmos, debemos tener claros los cimientos. Imagina que un algoritmo es como una receta de cocina: tienes una serie de pasos específicos que, si los sigues al pie de la letra, te darán un resultado esperado (el platillo).

Definición formal: Un algoritmo es un conjunto ordenado y finito de operaciones simples y bien definidas que permite resolver un problema o tomar una decisión.

Características esenciales de un algoritmo:

  • Preciso: Cada instrucción debe ser clara y no dar lugar a ambigüedades. No puede tener dobles sentidos.
  • Definido: Si se sigue un algoritmo dos veces con los mismos datos de entrada, se debe obtener el mismo resultado siempre. Es consistente.
  • Finito: Debe tener un número limitado de pasos. Si es infinito, ya no es un algoritmo, es un proceso (como el loop de un sistema operativo).

Partes de un algoritmo:

  • Entrada (Input): Los datos que necesita el algoritmo para comenzar (ej. harina, huevos, azúcar).
  • Proceso: Las operaciones y cálculos que se realizan con los datos de entrada (ej. batir, hornear).
  • Salida (Output): El resultado final obtenido (ej. el pastel).

2.2 Metodología de la programación

Aquí pasamos de la idea a la acción. No es solo sentarse a teclear código, sino seguir una metodología para construir software de calidad. La metodología más común y efectiva para resolver problemas se divide en fases:

  1. Análisis del problema: Entender qué se necesita resolver.
    • Leer el problema las veces que sea necesario.
    • Identificar los datos de entrada y el resultado esperado.
    • Preguntar: ¿Qué me piden? ¿Con qué datos cuento?
  2. Diseño del algoritmo: Definir cómo lo vamos a resolver.
    • Aquí es donde usamos herramientas como diagramas de flujo o pseudocódigo.
    • Se plantea la lógica y la secuencia de pasos.
  3. Codificación (Implementación): Traducir el algoritmo a un lenguaje de programación específico (Python, Java, C++, etc.).
  4. Prueba y depuración: Ejecutar el programa para verificar que funciona.
    • Prueba: Ejecutarlo con datos de prueba (válidos, inválidos, límites) para ver si da el resultado esperado.
    • Depuración: Corregir los errores (bugs) que aparecen.
  5. Mantenimiento: Mejorar el programa, agregar nuevas funciones o adaptarlo a nuevos entornos una vez que ya está en uso.

2.3 Diagramas de flujo

Un diagrama de flujo es la representación gráfica del algoritmo. Es como ver el mapa del tesoro. Usa símbolos estándar para que cualquier programador lo entienda.

Simbología principal (estándar ANSI):

  • Óvalo o Terminal: Indica el inicio o el fin del programa.
  • Paralelogramo o Datos (Entrada/Salida): Lectura de datos (input) o impresión de resultados (output).
  • Rectángulo o Proceso: Operaciones aritméticas y asignaciones (ej. Suma = A + B).
  • Rombo o Decisión: Indica una bifurcación. Tiene una pregunta, una condición, y dos o más caminos de salida (Sí/No, Verdadero/Falso).
  • Flechas o Líneas de flujo: Indican la dirección y el orden en que se ejecutan las instrucciones.

Ventajas:

  • Visual, fácil de entender la lógica general.
  • Facilita la detección de errores lógicos (ej. un ciclo que no termina).

Desventaja:

  • Si el algoritmo es muy complejo, el diagrama se vuelve enorme y difícil de manejar.

2.4 Seudocódigos

Es el paso intermedio entre el lenguaje humano y el lenguaje de programación. Es una mezcla de lenguaje natural (español, inglés) y elementos de programación. No se ejecuta en una computadora, pero es la base perfecta para escribir el código real.

Ventajas:

  • Ocupa menos espacio que un diagrama de flujo.
  • Es más cercano al lenguaje de programación final.
  • Es independiente del lenguaje de programación. El mismo pseudocódigo puede servir para pasarlo a Java o a Python.

Estructura típica de un pseudocódigo:

Algoritmo NombreDelAlgoritmo
    // Definición de variables
    Definir variable1, variable2 Como TipoDeDato (Entero, Real, Carácter, Lógico)

    // Entrada de datos
    Escribir "Mensaje para el usuario"
    Leer variable1

    // Proceso (Cálculos y Lógica)
    variable2 <- variable1 * 2

    // Estructuras de control
    Si (condicion) Entonces
        // Instrucciones si se cumple
    Sino
        // Instrucciones si no se cumple
    FinSi

    // Salida de datos
    Escribir "El resultado es: ", variable2
FinAlgoritmo
